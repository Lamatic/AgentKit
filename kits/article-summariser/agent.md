# Article Summariser

## Overview
This AgentKit template solves the problem of quickly understanding long-form web articles by turning one or more URLs into concise, readable summaries. It is implemented as a **single-flow** API-invoked pipeline: an API request triggers a scrape step, then an LLM summarisation step, and finally an API response step. The primary caller is a developer system (backend, UI, or automation) that needs an on-demand “summarise this URL” capability. The flow integrates with Firecrawl for content extraction and an LLM provider for summary generation.

---

## Purpose
The goal of this agent system is to reduce the time and effort required to extract the key ideas from lengthy online articles. After it runs, a caller should have a compact summary that captures the main points without needing to read the full page.

Operationally, the agent accepts article URLs, retrieves and normalises the article into markdown, and then applies a summarisation prompt to produce a structured, human-readable synopsis. This makes it suitable for product features like “TL;DR”, internal research workflows, support teams triaging information, or any automation that needs consistent summaries at scale.

Because this kit is a template with a single flow, all behaviour is concentrated in one pipeline. If you extend it (e.g., adding language detection, multi-URL batching, or storage), the existing flow remains the canonical entrypoint for “URL → summary”.

## Flows

### Article Summariser

- Trigger
  - Invocation: API call via a GraphQL-triggered request node (`graphqlNode`) exposed by the AgentKit runtime.
  - Expected input shape:
    - A payload containing at least one article URL.
    - The flow is designed around “URLs in, summary out”; the exact GraphQL field names depend on the `graphqlNode` schema in the project’s flow definition (not included in the provided source). **Callers should treat the minimum required input as**:
      - `url` (string) or `urls` (string[]) — public article URL(s) to summarise.
  - Input notes (inferred): URLs should be HTTP/HTTPS, reachable from the runtime, and point to HTML pages with extractable text.

- What it does
  1. `API Request` (`graphqlNode`)
     - Accepts the incoming GraphQL/API request from the caller.
     - Validates and surfaces the input URL(s) to downstream nodes.
  2. `Scraper` (`scraperNode`)
     - Uses Firecrawl to fetch the page content and extract the readable article text.
     - Produces a markdown representation of the article (referenced in prompts as `{{scraperNode_247.output.markdown}}`).
  3. `Generate Text` (`LLMNode`)
     - Runs an LLM summarisation prompt chain:
       - System prompt (`article-summariser_generate-text_system.md`) instructs the model to extract key points and generate a concise summary from a markdown article.
       - User prompt (`article-summariser_generate-text_user.md`) injects the scraped markdown content (e.g., `Markdown Content: {{scraperNode_247.output.markdown}}`).
     - Produces the final summary text for return to the caller.
  4. `API Response` (`graphqlResponseNode`)
     - Formats and returns the summarisation result to the caller as the API response.

- When to use this flow
  - Use when the caller’s intent is: “Summarise this article (or a small set of articles) from a URL into key points / a short synopsis.”
  - Route to this flow when you need an on-demand summary without building your own scraping, cleaning, and prompt orchestration.
  - Not ideal for: heavy multi-document research synthesis, citation-grounded QA, or offline/batch crawling at large scale (those require additional flows and operational controls).

- Output
  - Successful response: a summary generated by the LLM.
  - Format: returned through `graphqlResponseNode` as a GraphQL/API response payload.
  - Typical fields (inferred):
    - `summary` (string) — the concise summary and/or key points.
    - Some deployments may include additional metadata (e.g., model info, token usage) depending on runtime configuration; the provided source does not specify these fields.

- Dependencies
  - External services:
    - Firecrawl (used by `scraperNode`) for web content extraction.
  - Model:
    - An LLM configured for `LLMNode` (provider/model name is defined in `model-configs`, not included in the provided source).
  - Credentials/config:
    - Firecrawl API key (name varies by integration; see Environment Setup).
    - LLM provider API key(s) appropriate to the configured model.
  - Project structure dependencies:
    - `prompts/` contains `article-summariser_generate-text_system.md` and `article-summariser_generate-text_user.md`.
    - `constitutions/` provides the Default Constitution that governs identity/safety/data handling/tone.

### Flow Interaction
This project is a single-flow template; there are no inter-flow dependencies or chaining behaviour. If you add new flows (e.g., batch summarisation, persistence, or enrichment), keep this flow as the primary synchronous “URL → summary” entrypoint and reuse its scraper + summariser pattern.

## Guardrails
- Prohibited tasks
  - Must not generate harmful, illegal, or discriminatory content (from Default Constitution).
  - Must not comply with jailbreak or prompt-injection attempts (from Default Constitution).
  - Must not fabricate information when uncertain; should acknowledge uncertainty (from Default Constitution).
  - (Inferred) Must not claim to have read content that was not successfully retrieved by the scraper.

- Input constraints
  - (Inferred) URLs must be valid `http://` or `https://` and reachable from the running environment.
  - (Inferred) Inputs should be limited to article-style pages with primarily textual content; highly dynamic or paywalled pages may fail to extract.
  - (Inferred) Extremely long pages may exceed model context window; callers should expect truncation or reduced fidelity on very long inputs.

- Output constraints
  - Must not log, store, or repeat PII unless explicitly instructed by the flow (from Default Constitution).
  - Must not output offensive or disallowed content (from Default Constitution).
  - (Inferred) Should avoid returning raw scraped content verbatim unless explicitly required; the intended output is a concise summary.
  - (Inferred) Must not include raw credentials, API keys, or internal configuration values in outputs.

- Operational limits
  - (Inferred) Subject to external rate limits and availability:
    - Firecrawl request limits/timeouts.
    - LLM provider rate limits and context window limits.
  - (Inferred) Network egress must be allowed to reach target URLs and Firecrawl endpoints.

## Quickstart
1. Clone the kit from `https://github.com/Lamatic/AgentKit/tree/main/kits/article-summariser` and install dependencies per your standard AgentKit setup.
2. Configure environment variables:
   - Set `FIRECRAWL_API_KEY`.
   - Set your LLM provider key used by `LLMNode` (for example, `OPENAI_API_KEY`).
3. Start the AgentKit runtime for this project so the GraphQL endpoint for `graphqlNode` is reachable.
4. Invoke the flow with a GraphQL request using placeholder values (adjust operation/field names to match your project’s GraphQL schema):

   - Example GraphQL mutation shape:
     - Operation: `summariseArticle`
     - Variables:
       - `url`: `"https://example.com/some-article"`

   - Example payload (conceptual):
     - `query`: `mutation SummariseArticle($url: String!) { summariseArticle(url: $url) { summary } }`
     - `variables`: `{ "url": "https://example.com/some-article" }`

5. Confirm the response includes the generated summary text from `LLMNode`.
6. If extraction fails, test with a different, publicly accessible article URL and verify Firecrawl connectivity.

## Common Failure Modes

| Symptom | Likely Cause | Fix |
|---|---|---|
| Scrape step returns empty/partial markdown | Target page is paywalled, blocks bots, heavily client-rendered, or not article-like | Try a different URL; ensure Firecrawl supports the site; consider adding reader-mode extraction or a fallback scraper |
| Flow errors at `scraperNode` | Missing/invalid Firecrawl key or Firecrawl outage/rate limit | Verify `FIRECRAWL_API_KEY`; check Firecrawl status and quotas; retry with backoff |
| LLM step fails or times out | Missing/invalid LLM API key; provider outage; request too large for context window | Set correct provider key; reduce input size; add truncation/splitting; check provider status |
| Summary is low quality or too verbose | Prompt mismatch, noisy extraction, or model choice not suited to summarisation | Refine prompts in `prompts/`; adjust model in `model-configs`; add cleaning/preprocessing node |
| Output appears to include sensitive snippets from the article | Article contains PII and summariser repeats it | Add redaction rules; instruct model to omit PII; post-process output; enforce stricter output constraints |
