# Teams Assistant
A Microsoft Teams bot flow that receives direct bot-chat messages, retrieves relevant knowledge from the configured vector index, and replies with a grounded answer as the Teams-facing assistant entry point in the wider internal knowledge system.

## Purpose
This flow is responsible for handling conversational question answering inside Microsoft Teams. It listens for inbound bot-chat messages, uses the message text as a retrieval query against an already-populated vector database, and generates a response through a retrieval-augmented generation step. Its job is not to ingest or index knowledge; instead, it turns previously indexed internal content into a usable Teams support experience for employees.

The outcome of the flow is a contextual Teams reply based on the most relevant indexed content available in the selected vector store. That outcome matters because it gives end users a familiar collaboration interface for asking internal knowledge questions without leaving Teams. In practice, this makes the organization’s indexed documents, pages, and records accessible in a conversational format.

Within the broader agent architecture, this flow sits on the retrieval-and-synthesis side of the pipeline. Upstream indexation flows are expected to have already collected enterprise content, chunked it, embedded it, and stored it in the vector database. This flow then performs the runtime retrieval step and the final answer synthesis step for the Microsoft Teams channel. It is therefore a channel-specific assistant flow parallel to sibling assistant flows such as web chat or Slack, rather than a general ingestion or orchestration flow.

## When To Use
- Use when employees need to ask questions from within Microsoft Teams and receive answers grounded in the organization’s indexed knowledge base.
- Use when a direct bot conversation in Teams should act as the entry point for an internal RAG assistant.
- Use when the relevant enterprise data has already been indexed into the configured vector database by one or more indexation flows.
- Use when the incoming interaction is a Teams bot-chat message and the desired behavior is to generate and send a reply automatically.
- Use when you want a Teams-native assistant experience instead of routing users to a separate web chatbot or another messaging platform.

## When Not To Use
- Do not use when the knowledge base has not yet been indexed; this flow assumes the vector store already contains usable embeddings and content.
- Do not use for ingestion, crawling, chunking, or vectorization tasks; those responsibilities belong to the kit’s indexation flows.
- Do not use when the user interaction originates from Slack or a web chat widget; the sibling assistant flow for that channel is the correct choice.
- Do not use when Teams credentials have not been configured, because both the trigger and the outbound reply depend on authenticated Teams access.
- Do not use when the incoming payload is not a Teams bot-chat event or does not contain message text suitable for retrieval.
- Do not use when you need group-channel routing logic beyond this flow’s configured conversation mode; this implementation is configured around Teams bot chat and replies using the triggering conversation context.

## Dependencies
### Upstream Flows
This is a channel entry-point flow for runtime question answering, so no upstream flow invokes it directly as part of the execution graph.

However, it does depend on prior indexation flows having run successfully before it can produce useful answers. In the broader bundle, those ingestion flows must have already:
- Collected source content from enterprise systems such as web sources, cloud drives, SharePoint, OneDrive, S3, Postgres, or similar configured sources
- Normalized and chunked that content
- Embedded the chunks
- Stored them in the vector database selected by `RAGNode_233`

At runtime, this flow consumes the following upstream data produced outside the flow itself:
- Indexed vector data in the selected `vectorDB`
- Teams bot event fields from the trigger, especially `triggerNode_1.output.text`, `triggerNode_1.output.id`, and `triggerNode_1.output.conversation.id`

### Downstream Flows
No downstream Lamatic flow is defined in this execution graph. This flow terminates by sending a message back to Microsoft Teams.

Operationally, the response is consumed by the end user inside Teams. If an external orchestration layer monitors connector responses, it would most likely rely on the generated answer content and Teams delivery metadata, but no explicit downstream flow dependency is declared here.

### External Services
- Microsoft Teams — receives inbound bot-chat events and delivers the assistant’s reply — requires Teams `credentials`
- Vector database — stores and serves semantically searchable knowledge chunks for retrieval — requires the private `vectorDB` selection on `RAGNode_233`
- Embedding model — converts the inbound Teams query into vector space for similarity search — requires `embeddingModelName`
- Generative model — synthesizes the final answer from retrieved context and prompts — requires `generativeModelName`
- Prompt resources — provide the system and user instructions that shape RAG behavior — referenced by `teams_assistant_rag_system` and `teams_assistant_rag_user`
- Model configuration resource — supplies retrieval/generation settings such as `limit`, `memories`, `messages`, and `certainty` — referenced by `teams_assistant_rag`

### Environment Variables
- None explicitly declared in the flow source — this flow uses private configured inputs rather than named environment variables — used by `triggerNode_1`, `RAGNode_233`, and `teamsNode_784` through Lamatic-managed configuration

## Node Walkthrough
1. `Teams` (`triggerNode`) starts the flow when a Microsoft Teams bot-chat event is received. It is configured with `conversationType` set to `botChat`, so the flow is specifically listening for direct bot conversation traffic. From the incoming event, it exposes the user’s message text as `triggerNode_1.output.text`, along with reply-routing metadata such as the message `id` and `conversation.id`.

2. `RAG` (`dynamicNode`) takes the inbound Teams message text and uses it as the retrieval query through `queryField` mapped from `{{triggerNode_1.output.text}}`. It searches the configured `vectorDB` using the selected `embeddingModelName`, applies retrieval settings from `@model-configs/teams-assistant_rag.ts`, and then generates an answer using the selected `generativeModelName`. The answer is shaped by the referenced system and user prompts and returned as `RAGNode_233.output.modelResponse`.

3. `Teams` (`dynamicNode`) sends the generated answer back to Microsoft Teams. Its `message` field is mapped from `{{RAGNode_233.output.modelResponse}}`, and it uses the trigger-derived `conversation.id` and message `id` to route the reply into the originating conversation context. The node is configured for Teams send-message behavior and includes `includePoweredBy` enabled.

4. `addNode` (`addNode`) is a canvas placeholder rather than an operational processing step. It does not contribute business logic or transform flow data.

## Error Scenarios
| Symptom | Likely Cause | Recommended Fix |
|---|---|---|
| The flow never triggers on Teams messages | Teams `credentials` are missing, invalid, or the bot is not correctly connected to Microsoft Teams | Reconfigure the Teams credential on `teamsNode_784`, verify the trigger inherited it correctly on `triggerNode_1`, and confirm the bot is installed and permitted to receive direct bot-chat events |
| The bot triggers but returns no useful answer | The vector database is empty, the wrong `vectorDB` was selected, or upstream indexation flows have not populated the knowledge base | Run the appropriate indexation flow first, verify content exists in the selected vector store, and confirm `RAGNode_233` points to the intended database |
| The assistant responds with irrelevant content | The selected embedding or generative model is poorly matched to the data, retrieval settings are misconfigured, or indexed content quality is weak | Review `embeddingModelName`, `generativeModelName`, and the referenced model config; re-index cleaner content if necessary |
| The flow fails at the RAG step | Required private inputs for `vectorDB`, `embeddingModelName`, or `generativeModelName` were not configured | Populate all required private inputs on `RAGNode_233` before deployment |
| The bot cannot send the reply back to Teams | Outbound Teams credentials are invalid, the conversation context is missing, or the conversation/message identifiers from the trigger are malformed | Validate Teams authentication, inspect the trigger payload for `conversation.id` and `id`, and confirm the bot has permission to post into the conversation |
| A user asks a question but receives an incomplete or truncated answer | Model token limits, retrieval limits, or prompt constraints reduced the amount of context or output length | Adjust the referenced model configuration, review retrieval `limit` behavior, and test with a model that supports the required context window |
| The flow is deployed but answers are stale or missing recent documents | Upstream indexation flows have not run recently or new source content was never embedded into the vector store | Re-run the relevant ingestion/indexation flow and confirm the new content was chunked, embedded, and stored in the same database this assistant queries |
| Non-Teams invocations fail or produce malformed inputs | This flow expects a Teams bot-chat trigger and is not designed for arbitrary API-style payloads | Route non-Teams traffic to the appropriate assistant flow, such as the web or Slack variant, instead of reusing this trigger |

## Notes
- This flow is tightly channel-bound: it is the Microsoft Teams assistant surface for the internal knowledge system, not a generic reusable RAG endpoint.
- The trigger is configured for `botChat`, which indicates a direct bot conversation pattern rather than a broad multi-channel or generalized webhook intake.
- The reply node uses both `conversationId` and `channelMessageId` from the trigger context, so successful response delivery depends on preserving the event metadata emitted by Teams.
- The RAG node’s detailed retrieval behavior is partly externalized into `@model-configs/teams-assistant_rag.ts`; operators should review that resource when tuning retrieval depth, memory behavior, or answer style.
- Prompt behavior is also externalized. Changes to `@prompts/teams-assistant_rag_system.md` and `@prompts/teams-assistant_rag_user.md` can materially change grounding, tone, and response structure without altering the flow graph.
- Because the flow uses private configuration inputs for credentials and model selection, portability between environments depends on correctly re-binding those private values after import or deployment.
