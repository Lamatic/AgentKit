# Firecrawl Webhook
A webhook-driven ingestion flow that receives crawler page events, extracts indexable page content, and writes embeddings into a vector database as the canonical ingestion path for this kit.

## Purpose
This flow is responsible for converting incoming crawl callbacks into searchable knowledge. Its specific job is to accept webhook payloads from an external crawler, check whether the event represents a page that should be indexed, reshape the payload into content-plus-metadata form, generate embeddings from the extracted text, and persist those embeddings to a vector store.

The outcome is a vector-indexed representation of crawled pages that downstream retrieval systems can search semantically. That matters because the broader agent pipeline depends on having clean, content-focused documents available for retrieval; without this ingestion step, crawled data remains raw event payloads rather than usable knowledge assets.

Within the wider system, this flow sits at the ingestion boundary between crawling and retrieval. It is an entry-point flow triggered directly by an external orchestrator rather than by another Lamatic flow. In a crawl-to-retrieve chain, it performs the normalization, embedding, and indexing stages so later query-time systems can retrieve relevant passages from the indexed corpus.

## When To Use
- Use when an external crawler posts a webhook event for a newly crawled or updated page.
- Use when the incoming payload represents a `crawl.page` event that should be transformed into vector-searchable knowledge.
- Use when you need to keep a vector index synchronized with crawler output using overwrite semantics for duplicates.
- Use when this kit is acting as the ingestion bridge between a crawling system and a semantic retrieval system.
- Use when page content and metadata are available in the webhook payload, or can be derived by the flow's `Code` step.

## When Not To Use
- Do not use when the incoming webhook payload is not a page-level crawl event; the flow only routes indexing logic for `crawl.page` events.
- Do not use when you need query-time retrieval, answer generation, or synthesis; this flow only ingests and indexes content.
- Do not use when the source data is not arriving through the webhook trigger; this flow has no declared manual inputs.
- Do not use when embedding-model or vector-store configuration has not been set up in the Lamatic environment.
- Do not use when your crawler emits a materially different schema and the `Code` script has not been updated to map it into the expected `vectorData` and `MetaData` outputs.
- Do not use for delete or retraction workflows unless you extend the logic, because the current implementation only overwrites duplicates rather than handling removals explicitly.

## Inputs
| Field | Type | Required | Description |
|---|---|---|---|
| `type` | `string` | Yes | Event type from the webhook payload. The flow checks this field and only indexes events whose value is `crawl.page`. |
| Webhook request body | `object` | Yes | Full crawler payload delivered to the `Webhook` trigger. The `Code` node relies on this payload to extract page content and metadata for indexing. |
| Page content fields | `string` or `object` | Conditional | Content-bearing fields within the webhook body, such as extracted text, markdown, or other page-body representations expected by the transformation script. |
| Page metadata fields | `object` | Conditional | Metadata associated with each crawled page, such as URL, title, timestamps, or crawl attributes, if present in the source payload. |

The trigger has no statically declared Lamatic `inputs`, so all runtime input arrives through the webhook body. The only explicit condition visible in the flow definition is that `triggerNode_1.output.type` must equal `crawl.page` for indexing to proceed. The exact payload schema beyond that is determined by the external crawler and the script referenced at `@scripts/firecrawl-webhook_code.ts`; if the payload shape changes, that script must be updated accordingly.

## Outputs
| Field | Type | Description |
|---|---|---|
| HTTP/webhook response | `object` | Terminal response emitted by the flow after either the no-op branch or the indexing branch completes. |
| `vectors` | `array` | Embedding vectors generated by `Vectorize` from the transformed page content. Used internally by the `Index` node. |
| `MetaData` | `object` or `array` | Structured metadata produced by the `Code` node and passed into `Index` for storage alongside vectors. |
| `vectorData` | `string` or `array` | Text payload prepared by the `Code` node for embedding generation. Used internally by `Vectorize`. |

Externally, this flow behaves like a webhook-processing endpoint that returns after execution completes. Internally, its meaningful product is the indexed vector-store write rather than a rich API payload. The exact terminal response body is not explicitly defined in the exported flow source because the two unnamed `addNode` steps likely act as response terminators or pass-through nodes, so consumers should treat the webhook success/failure status as the primary execution signal unless they inspect runtime node outputs in Lamatic.

## Dependencies
### Upstream Flows
- None. This is the entry-point flow for this template and is invoked directly by an external crawler or orchestration system through the `Webhook` trigger.
- The practical upstream dependency is the external crawling process, which must have already fetched page data and sent a webhook payload containing at least a `type` field and the content needed for transformation.

### Downstream Flows
- No separate Lamatic flows are defined downstream in the provided kit context.
- Operationally, downstream consumers are retrieval or question-answering systems that query the vector database populated by this flow, but they do not consume this flow's response payload directly.

### External Services
- External crawler webhook source — sends crawl event payloads into the `Webhook` node — no credential is configured in this flow itself because it is an inbound trigger.
- Embedding model provider — converts `vectorData` into embeddings in the `Vectorize` node — required credentials depend on the embedding provider configured in Lamatic.
- Vector database / index backend — stores vectors and metadata in the `Index` node — required credentials depend on the configured vector-store connector.
- Script resource `@scripts/firecrawl-webhook_code.ts` — custom transformation logic used by the `Code` node to extract content and metadata — no separate credential implied by the flow definition.

### Environment Variables
- `EMBEDDING` provider credentials — required to generate embeddings — used by the `Vectorize` node.
- `VECTOR DATABASE` connector credentials — required to write vectors and metadata to the index — used by the `Index` node.
- Any crawler verification secret configured at deployment time — if webhook authentication is enforced outside the exported flow definition — used by the `Webhook` node or deployment layer.

## Node Walkthrough
1. `Webhook` (`webhookTriggerNode`) receives the inbound HTTP request from the crawler or orchestration system. This is the runtime entry point for the flow, and all payload data enters here because the flow declares no separate static inputs.

2. `Condition` (`conditionNode`) inspects `triggerNode_1.output.type` and compares it to `crawl.page`. Only page-level crawl events are intended to move into the indexing path. Although the edge labels in the export appear mismatched relative to the condition names, the logical intent from the condition definition is clear: page events proceed to processing, while all other event types short-circuit.

3. The non-indexing branch goes to the first unnamed `addNode` (`addNode_728`). This branch is effectively the fallback path for events that do not qualify for indexing. No field assignments are shown in the export, so it most likely acts as a simple placeholder or response-shaping step.

4. Both execution paths terminate through the second unnamed `addNode` (`addNode_817`). On the fallback route, it receives control directly from `addNode_728`; on the indexing route, it receives control after indexing completes. Because it is the shared terminal node, it likely represents the final response stage for the webhook invocation.

5. On the indexing branch, `Code` (`codeNode_474`) runs the script at `@scripts/firecrawl-webhook_code.ts`. This is the transformation heart of the flow: it extracts only the page content needed for embedding and prepares a metadata structure for storage. The node emits at least two named outputs consumed later: `vectorData` for embedding input and `MetaData` for index metadata.

6. `Vectorize` (`vectorizeNode_657`) takes `{{codeNode_474.output.vectorData}}` as `inputText` and generates embeddings. The embedding model is not hard-coded in the export, so it must be configured through the Lamatic environment or node settings at deploy time.

7. `Index` (`IndexNode_117`) writes the generated vectors to the configured vector database using `{{vectorizeNode_657.output.vectors}}` as the vector payload and `{{codeNode_474.output.MetaData}}` as the metadata payload. The node is configured with `duplicateOperation` set to `overwrite`, meaning repeated ingestions of the same logical record replace existing indexed content rather than creating uncontrolled duplicates.

8. After indexing completes, control passes to the shared terminal `addNode_817`, allowing the webhook request to finish cleanly after either successful indexing or no-op handling.

## Error Scenarios
| Symptom | Likely Cause | Recommended Fix |
|---|---|---|
| Webhook call succeeds but nothing is indexed | The incoming `type` is not `crawl.page`, so the flow takes the fallback branch | Inspect the webhook payload and ensure the event type matches the condition, or update the condition logic if your crawler uses a different event name |
| Flow fails in `Code` | The webhook payload schema does not match what `@scripts/firecrawl-webhook_code.ts` expects | Compare actual crawler payloads against the transformation script and update field mappings for content and metadata extraction |
| `Vectorize` fails with authentication or provider errors | Embedding provider credentials are missing or invalid | Configure the required embedding-model credentials in Lamatic and verify the embedding model selection for the `Vectorize` node |
| `Index` fails with connector or authorization errors | Vector database credentials or connector settings are missing or incorrect | Reconfigure the vector-store integration, verify network access, and confirm the target index exists and is writable |
| Indexed records contain empty or poor-quality content | The crawler payload lacks extracted text, or the transformation script is selecting the wrong field | Ensure the crawler sends usable page content and revise the `Code` script to choose the correct content source |
| Duplicate pages do not preserve historical versions | The node is configured with `duplicateOperation` set to `overwrite` | Change duplicate handling if version retention is required, or include version identifiers in metadata and primary key strategy |
| Flow returns an unexpected minimal response | The terminal `addNode` steps have no visible configured fields in the export | Explicitly configure response shaping if callers need a structured acknowledgment body beyond status success |
| Upstream crawler appears to have run, but this flow never triggers | The external crawler is not posting to the deployed webhook endpoint, or webhook authentication/routing is misconfigured | Verify the deployed webhook URL, delivery logs, and any ingress authentication or signature validation configured outside the flow export |
| Index writes fail or behave unpredictably for duplicate content | `primaryKeys` is empty in the `Index` node, so record identity may depend on connector defaults or metadata structure | Define a stable primary key strategy, typically based on URL or canonical page identifier, to make overwrite behavior deterministic |

## Notes
- The exported flow references a custom script at `@scripts/firecrawl-webhook_code.ts`, but the script body is not included here. That script defines the practical contract for how incoming crawler payloads are converted into `vectorData` and `MetaData`.
- The condition edges in the exported graph appear labeled inconsistently relative to the condition definitions. Documentation here follows the logical intent expressed by the condition itself: only `crawl.page` events should proceed to indexing.
- `embeddingModelName` is empty in the `Vectorize` node export, which means deployment-time configuration is important. **Do not assume a default embedding model is available in every environment.**
- `primaryKeys` is empty in the `Index` node export. This can be acceptable in some connectors, but for reliable overwrite semantics you should usually configure a stable identity field such as a canonical URL.
- This flow is optimized for ingestion, not retrieval. Its success criteria are correct transformation and durable indexing, not user-facing response richness.
- Because both successful and skipped executions converge on the same terminal node, observability should rely on node-level execution traces and index write metrics rather than only on the final webhook response.
