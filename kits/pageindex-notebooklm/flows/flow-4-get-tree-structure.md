# Flow 4 Get Tree Structure
This flow retrieves a stored document tree for browsing or deletes a document record, providing the operational tree-management endpoint in the wider PageIndex document lifecycle.

## Purpose
This flow is responsible for a narrow but important operational task in the PageIndex system: looking up the persisted hierarchical tree for a specific document, or removing that document from storage when the caller intends to delete it. In this kit, the tree is the core structural index used instead of vector embeddings, so being able to fetch that tree reliably is how the frontend can render a navigable document outline and expose document metadata such as file name, creation time, and node count.

The flow produces one of two outcomes from the same API surface. For `get_tree`, it returns the stored `tree` plus related document metadata from the `documents` table. For any other `action` value, it follows the delete branch and issues a delete against the same table, returning confirmation details for the removed document. This matters because the broader agent pipeline is not only about retrieval and answering; it also needs lightweight operational endpoints so users can inspect indexed documents and manage their lifecycle.

Within the broader system described by the parent agent, this flow sits after ingestion and outside the answer-generation path. Flow 1 creates and saves the document structure; this flow exposes that saved structure for browsing and maintenance. It does not plan retrieval, synthesize answers, or build indexes. Instead, it serves the browse-and-admin layer of the pipeline, alongside the list-documents flow, so the frontend can inspect what has already been indexed before or instead of invoking the chat flow.

## When To Use
- Use when the frontend needs the full stored hierarchy for a known `doc_id` so it can render a tree viewer or document outline.
- Use when a user selects an existing indexed document and wants to inspect its section structure before asking questions.
- Use when the system needs document-level metadata associated with the stored tree, including `file_name`, `tree_node_count`, and `created_at`.
- Use when a user explicitly requests deletion of a previously indexed document and the caller can provide its `doc_id`.
- Use when an orchestration layer needs a lightweight document-management endpoint rather than a retrieval or answer-generation flow.

## When Not To Use
- Do not use for uploading or indexing a new PDF; that is handled by Flow 1, which creates the document record and persisted tree.
- Do not use for answering questions against a document; the chat and retrieval path belongs to Flow 2.
- Do not use for listing all indexed documents; the sibling browse endpoint for that is Flow 3.
- Do not use if the caller does not know the target `doc_id`; this flow operates on a single existing document record.
- Do not use if the request payload is a file upload or multipart document input rather than a simple API request with scalar fields.
- Do not use if the database credentials for the Postgres connector have not been configured; both branches depend on a working database connection.

|---|---|---|---|
| `doc_id` | `string` | Yes | Unique document identifier used to look up or delete a row in the `documents` table. |
| `action` | `string` | Yes | Operation selector. `get_tree` triggers tree retrieval; any other value falls through to the delete branch. |

The trigger schema expects both fields as strings. The flow assumes `doc_id` matches an existing row in the `documents` table. The only explicitly recognized action is `get_tree`; because the condition node uses an `Else` branch for everything else, any non-`get_tree` value will attempt deletion. Callers should therefore treat `action` as effectively constrained to `get_tree` or an intentional delete command generated by trusted application logic.

The API response is a structured JSON object assembled by the `Merge Response` code node and then mapped directly to the response node. On the retrieval path, the response includes the full stored tree payload plus metadata from the matching document row. On the delete path, fields related to the tree may be absent or null, depending on how the merge script normalizes the delete result. Completeness therefore depends on both the branch taken and whether the target document existed in the database.

## Dependencies
### Upstream Flows
- `flow-1-upload-pdf-build-tree-save` must have run previously for `get_tree` to return meaningful data. That flow is responsible for ingesting the PDF, building the hierarchical tree, and persisting the document row that this flow reads from.
- From that upstream ingestion flow, this flow implicitly depends on a persisted `doc_id` and the existence of stored document fields in the `documents` table such as `tree`, `file_name`, `tree_node_count`, and `created_at`.
- This is not itself an ingestion entry point; it is an operational follow-up flow for already indexed documents.

### Downstream Flows
- No core downstream Lamatic flow is shown as consuming this flow directly.
- In practice, the primary consumer is the frontend or an orchestration layer that uses `tree`, `file_name`, `tree_node_count`, and `created_at` to render document-management and tree-browsing interfaces.
- The chat flow does not need this flow’s API response directly; it reads from persisted storage as part of its own retrieval path.

### External Services
- Supabase PostgreSQL — stores and serves document rows, including the persisted tree, and executes document deletion — Postgres connector credentials selected on `postgresNode_206` and `postgresNode_113`

### Environment Variables
- None explicitly declared in the flow source — database access is supplied through Lamatic credential bindings on `Get Tree` and `Delete Document`

## Node Walkthrough
1. `API Request` (`graphqlNode`) receives a realtime API call with two scalar fields: `doc_id` and `action`. This is the public trigger surface for the flow.
2. `Condition` (`conditionNode`) inspects `{{triggerNode_1.output.action}}`. If the value is exactly `get_tree`, execution routes to the retrieval branch. If it is anything else, execution routes to the delete branch.
3. `Get Tree` (`postgresNode`) runs on the `get_tree` branch. It queries the `documents` table for the row whose `doc_id` matches the trigger input and selects `tree`, `file_name`, `tree_node_count`, and `created_at`, limited to one record.
4. `Delete Document` (`postgresNode`) runs on the `Else` branch. It deletes the row in `documents` whose `doc_id` matches the trigger input and returns the deleted row’s `doc_id` and `file_name` for confirmation.
5. `Merge Response` (`codeNode`) receives whichever branch executed and normalizes the database result into a single response shape. This node is where branch-specific outputs are converted into the shared fields exposed by the API response.
6. `API Response` (`graphqlResponseNode`) maps normalized fields from `codeNode_merge.output` into the final JSON response sent back to the caller, including `success`, `action`, `message`, `doc_id`, `tree`, `file_name`, `tree_node_count`, and `created_at`.

## Error Scenarios
| Symptom | Likely Cause | Recommended Fix |
|---|---|---|
| Database node fails before returning data | Missing or invalid Postgres credentials on `postgresNode_206` or `postgresNode_113` | Configure the correct Lamatic Postgres credential binding for both database nodes and verify the target Supabase instance is reachable. |
| Response indicates no tree or returns null document fields | `doc_id` does not exist in the `documents` table, or Flow 1 never created the record | Confirm the document was successfully ingested and persisted by Flow 1, then retry with the correct `doc_id`. |
| A delete occurs when the caller expected retrieval | `action` was not exactly `get_tree`, so the condition fell into the `Else` branch | Restrict callers to trusted action values and validate `action` before invoking the flow. If necessary, harden the branch logic to require an explicit delete token. |
| Retrieval returns empty data even though the document exists | The stored row may be incomplete, or the query is pointed at the wrong database/schema through credentials | Validate the underlying `documents` row in Supabase and ensure the configured credential targets the expected environment. |
| API request is rejected or behaves unpredictably | Input payload is malformed, missing `doc_id`, or missing `action` | Send both trigger fields as strings in the request body and ensure the payload matches the declared schema. |
| Upstream tree-dependent UI cannot display structure | Flow 1 ingestion failed to produce or persist `tree` and related metadata | Re-run the ingestion/indexing flow and verify that `tree`, `tree_node_count`, and `file_name` were saved correctly before calling this flow. |

## Notes
- The branch design is intentionally compact but operationally risky: only `get_tree` is explicitly matched, and every other action value routes to deletion. **Do not expose arbitrary caller-controlled action strings without upstream validation.**
- Deletion is implemented against the `documents` table only in the exported flow source. If associated records exist in other tables, their cleanup depends on database-level cascading rules or external data-model design rather than this flow itself.
- The flow’s final response contract depends partly on the referenced merge script `@scripts/flow-4-get-tree-structure_merge-response.ts`. The response node shows the expected output fields, but message wording and exact success semantics are determined there.
- This flow is optimized for operational simplicity rather than rich validation. It assumes the caller already has a valid `doc_id` from document ingestion or listing flows.
