# Webpage QA
This flow answers a user’s question about a live webpage by scraping the page, grounding an LLM on the extracted content, and returning the answer as the system’s entry-point QA capability for webpage content.

## Purpose
This flow is responsible for turning a caller-provided webpage into an on-demand question answering source. It solves the specific sub-task of fetching readable content from a public URL, converting that content into a form suitable for language-model reasoning, and producing an answer to a user’s question without requiring a prebuilt index or manual content preparation.

The outcome is a single grounded answer returned through an API response. That matters because it gives the wider agent system a lightweight way to answer questions about arbitrary pages in real time, which is useful when content is dynamic, one-off, or not worth ingesting into a persistent retrieval layer. The value of the flow is its low setup overhead: provide a `url` and a `question`, and the flow handles acquisition and synthesis end to end.

In the broader chain described by the parent agent, this flow acts as the complete request-to-answer path for webpage QA. It combines the retrieval step and the synthesis step inside one linear pipeline: the `Scraper` node retrieves page content, and the `Generate Text` node synthesizes an answer from that content. Because the trigger is an API request and there are no prerequisite flows, it serves as a standalone entry-point flow that can also be embedded by external applications needing “ask this page” behavior.

## When To Use
- Use when a caller wants an answer grounded in the contents of a specific public webpage.
- Use when the request includes a valid webpage `url` and a natural-language `question` about that page.
- Use when you need live or near-live page content rather than answers from a pre-indexed knowledge base.
- Use when you want a simple synchronous API pattern: submit request, scrape page, generate answer, return response.
- Use when the page is primarily text-based and the important information is likely present in the main rendered content.
- Use when embedding webpage question answering into an app, product workflow, support surface, or internal tool without building separate retrieval infrastructure.

## When Not To Use
- Do not use when the caller cannot provide a valid absolute webpage URL.
- Do not use when the question is unrelated to the supplied page content and requires broader web search, multiple sources, or internal enterprise data.
- Do not use when the target content is behind authentication, paywalls, CAPTCHAs, IP restrictions, or other controls the scraper cannot access.
- Do not use when the target page is highly interactive and the needed content only appears after complex client-side actions not covered by the configured scrape.
- Do not use when you need structured extraction, field-by-field parsing, or deterministic schema output rather than a natural-language answer.
- Do not use when a sibling or alternate flow is intended to answer from a curated document store, vector index, database, or another domain-specific retrieval source.
- Do not use when scraping credentials have not been configured, because the flow depends on an external scraping service.

## Inputs
| Field | Type | Required | Description |
|---|---|---|---|
| `url` | `string` | Yes | Absolute URL of the webpage to scrape and use as the grounding source for the answer. |
| `question` | `string` | Yes | Natural-language question the model should answer using the scraped webpage content. |

The trigger source file does not declare an explicit schema, but the flow wiring and parent agent context make `url` and `question` the required logical inputs. The `url` is assumed to be a fetchable public web address. The `question` should be clear, specific, and scoped to information that could plausibly be found on the supplied page. There is no explicit length validation in the flow definition, so callers should avoid excessively long prompts or malformed URLs and should assume normal downstream service limits apply.

## Outputs
| Field | Type | Description |
|---|---|---|
| `answer` | `string` | Natural-language answer generated by the LLM from the scraped webpage content. |

The response is a simple object containing one prose field, `answer`. The output is not a citation bundle, extract list, or structured schema unless the underlying prompt has been customized to behave that way. Completeness depends on scrape quality and model behavior: if the page content is partial, noisy, or missing the requested information, the returned answer may be incomplete or may reflect the model’s uncertainty.

## Dependencies
### Upstream Flows
- None. This is a standalone entry-point flow invoked directly through an API request.
- The flow does not consume outputs from any prior Lamatic flow; it expects the caller to provide `url` and `question` at invocation time.

### Downstream Flows
- None are defined in the provided kit context.
- This flow most commonly serves as a terminal request-response operation, returning `answer` directly to the invoking application or orchestration layer.

### External Services
- Firecrawl scraping service — fetches and extracts the target webpage into usable content — required credential or environment variable: `FIRECRAWL_API_KEY`
- Configured LLM via Lamatic model configuration — generates the grounded answer from scraped content and the system prompt — required credential or environment variable: depends on the referenced model configuration in `@model-configs/webpage-qa_generate-text.ts`
- Lamatic API trigger/response runtime — receives inbound request data and returns the flow response — credential handled by the deployment/runtime environment

### Environment Variables
- `FIRECRAWL_API_KEY` — authenticates the webpage scraping request — used by the `Scraper` node

## Node Walkthrough
1. `API Request` (`graphqlNode`) receives the inbound API call that starts the flow. In practice, the request is expected to include at least `url` and `question`. The node acts as the entry point and exposes request values for downstream node mappings.

2. `Scraper` (`scraperNode`) reads `url` from `{{triggerNode_1.output.url}}` and fetches the target webpage through the configured scraping integration. It is set to scrape desktop content with `mobile` disabled, wait briefly for page readiness, and prefer `onlyMainContent`, which aims to reduce chrome, navigation, and peripheral page noise. The result is extracted page content for downstream reasoning, typically as markdown or equivalent readable text.

3. `Generate Text` (`LLMNode`) runs the referenced system prompt `@prompts/webpage-qa_generate-text_system.md` with the referenced model configuration `@model-configs/webpage-qa_generate-text.ts`. Although the direct variable wiring is abstracted into the prompt and model config references, the intended behavior from the kit context is clear: the model is grounded on the scraper output and answers the caller’s `question` using the scraped page as the source of truth. This node produces `generatedResponse` as the final answer text.

4. `API Response` (`graphqlResponseNode`) maps `{{LLMNode_449.output.generatedResponse}}` into the response payload field `answer` and returns it to the caller. This is the terminal response for the request-response cycle.

## Error Scenarios
| Symptom | Likely Cause | Recommended Fix |
|---|---|---|
| Request fails before scraping begins | Missing or invalid `url` in the API request | Validate that `url` is present, non-empty, and a fully qualified public URL before invoking the flow. |
| Flow returns a weak or irrelevant answer | `question` is vague, too broad, or not actually answerable from the provided page | Rewrite the question to be specific and page-scoped; ensure the requested information exists on the target page. |
| Scraper node errors with authentication or access failure | `FIRECRAWL_API_KEY` is missing, invalid, or not available in the runtime environment | Set `FIRECRAWL_API_KEY` correctly for the deployed environment and verify the scraping integration is enabled. |
| Scraper returns little or no useful content | The page blocks scraping, has minimal main content, requires login, or depends on interactions not captured by the configured scrape | Test the target URL manually, use a publicly accessible page, or adjust the scraping strategy in a revised version of the flow. |
| LLM answer is empty or missing from the response | The model call failed, the prompt/config reference is misconfigured, or the scrape produced no grounding text | Check the referenced prompt and model configuration, confirm the model provider credentials are set, and verify scraper output is populated. |
| Response contains hallucinated or overconfident statements | The prompt/model grounding is insufficient relative to sparse or ambiguous webpage content | Strengthen the prompt to require uncertainty handling and “not found” behavior; ensure the page contains clear source material. |
| Invocation fails in an orchestrated pipeline expecting prior context | An upstream system assumed another flow had already prepared or validated inputs, but this flow is standalone | Invoke this flow directly with raw `url` and `question`, or add an upstream validation/preprocessing step in the orchestrator. |
| Malformed response expectations in client code | Client expects structured fields beyond `answer` | Update the client contract to consume the single `answer` field, or modify the flow’s response mapping if richer output is required. |

## Notes
- The flow source declares no explicit trigger schema in `inputs`, so input contract discipline must be enforced by the caller, deployment wrapper, or Studio configuration.
- The scraper is configured with `onlyMainContent: true`, which is generally good for QA quality but can omit useful sidebar, footer, or metadata content if that is where the answer lives.
- The scrape uses a fixed `waitFor` value of `123`, suggesting a short render delay. Pages with slower hydration or delayed content loading may produce incomplete extraction.
- No conditional routing or fallback behavior is defined. If scraping fails or produces poor content, the flow still has no alternative retrieval path.
- The exact model, message wiring, and memory behavior live in `@model-configs/webpage-qa_generate-text.ts`, so operational characteristics such as token limits, model family, and provider-specific credentials are determined there rather than in the flow file.
- The generated answer is returned as a single field with no built-in citations, confidence score, or raw scrape payload. If observability or explainability is important, consider extending the response mapping in a future revision.
